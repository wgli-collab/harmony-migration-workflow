## 零、审计前置约束（Audit Prerequisites）

以下规则和标准不是建议，是硬约束。违反任何一条都会导致漏报或错误判定。

**原则零**：代码的默认状态是"有问题的"，不是"没问题的"。审计者的工作是证明代码正确，而不是假设代码正确然后找反例。"逻辑不报错"不等于"功能完整"；"代码不崩"不等于"假设成立"。

### 审计评判标准（WHAT — 好的代码必须满足什么）

以下 6 条标准定义"好的代码应该满足什么"。执行纪律（R1-R18）定义"怎么审才不漏"，评判标准定义"审的时候以什么为基准"。标准被违反 = 代码有问题；纪律被违反 = 审计过程有问题。

| # | 标准 | 实例 / 边界 |
|---|------|------------|
| ① | **不隐式降级为默认行为** — 未知 method / 协议 / 参数值必须显式报错，不能自动退回默认 | 未知 HTTP method 不应默认 GET；不支持的协议不应默认 http |
| ② | **验证逻辑与实际运行路径一致** — 运行时代码路径接受的类型 / 值，验证器也必须接受，不能在验证层拒绝合法输入 | adapter 字段：运行时 `getAdapter()` 接受函数类型，但 `_VALIDATOR_SCHEMA` 只接受字符串 → 合法配置被验证器误拒 |
| ③ | **大小写敏感域统一处理** — Header 名、set-cookie、URL host 等域在所有代码路径中必须一致地做 case-insensitive，不能某些路径区分、另一些路径不区分 | `getSetCookie()` 直接 `_data.get(key)` 不走 `_findMapKey()` → 其他 header 方法大小写不敏感，只有它敏感 |
| ④ | **安全边界不可跳过** — 每个外部输入点（URL、header、body、config）必须经过安全校验，不能假设"这条路径应该安全"而跳过 | SSRF 过滤必须覆盖 private/loopback/localhost；`hasOwnProp()` 必须防原型污染；header 值必须过滤控制字符 |
| ⑤ | **兼容策略必须显式** — 旧 API 与新 API 的桥接、不同机制的共存，必须在代码中明确处理，不能靠隐式兼容 | CancelToken 与 AbortSignal 的组合语义必须在 `composeSignals` 中显式定义，不能靠"恰好能用" |
| ⑥ | **错误必须透明可追踪** — 失败时必须抛出类型化错误（含 code/status/message），不能抛出模糊 Error 或静默吞错 | `settle()` 必须根据 `validateStatus` 正确分类；网络错误 vs 取消错误 vs HTTP 状态错误必须可区分 |

### 执行纪律（HOW — 审计动作必须遵守什么）

#### 方法论层（防漏报）

> 以下实例源自 axios 迁移审计，作为通用模式的参考示例。迁移其他库时，替换为对应库的具体函数名。

| # | 规则 | 违反会导致 |
|---|------|-----------|
| **R1** | **每个函数（export + 私有）必须回答：谁调用了它？** — 用 `grep -r` 验证调用链。回答不了 = 死代码，不管是 export 还是模块内私有函数 | 把"代码存在且正确"当成"功能生效"（实例：export: settle / validator / transformData / composeSignals；私有: _simpleSerialize 0 次调用） |
| **R2** | **R1 确认 import 链后，必须比对 producer 的返回类型 / 接口是否满足 consumer 的调用约定** — `grep` 证明调用链存在，现在证明契约匹配 | 只验证第一跳 import，漏掉第二跳接口契约断裂（实例：dispatchRequest import composeSignals ✓ 但 ComposedSignal 缺少 subscribe() → adapter 取消失效） |
| **R3** | **类型定义 ↔ 消费代码 双向比对**：类型里每个字段 → 找哪行代码读它；代码里每个 `cfg['X']` → 回找类型声明 + 默认值。两个方向都验，不只验声明→代码 | 把"接口声明了"当成"代码实现了"（实例：auth / maxRedirects 等字段只声明无消费）；或反向——代码读了字段但类型没声明、默认值没提供（实例：transformRequest / transformResponse；ResponseType 三个源说法不一致） |
| **R4** | **每个接受复杂类型参数（object / Array / union / Record）的函数，必须列出原版对该参数支持的子类型与嵌套结构，逐类对照本版覆盖情况。覆盖不全 = 功能收窄，即使逻辑不报错也是 bug。** — `grep` 确认调用链后，逐参数比对输入空间 | 逻辑正确但覆盖范围比原版窄，审逻辑看不出（实例：buildURL 的 for 循环逻辑正确，但只处理标量和一维数组，嵌套对象 → `[object Object]`；ParamsSerializerOptions.dots / indexes / metaTokens 未实现。params 参数接受 `object` 类型，但只实现了浅层序列化） |
| **R5** | **每个 try / if 成功分支审完后，必须同步审查对应的 catch / else** — 成对审计，不同步不签字 | 成功路径正确、错误/else 路径遗漏（实例：headers `.toJSON()` 不对称 + rejected 拦截器链 3 处缺失 + customProps 静默丢弃只看了 status 路径 + validator 类型守卫只看了 function 路径） |
| **R6** | **每个来自外部源（SDK 返回值、用户配置、网络响应体）的属性访问链，必须验证路径上每一步都可能为 undefined / null / 类型不匹配** — 加守卫或注释说明前提条件。触发条件：属性访问的根对象来自 `import` 的外部模块返回值或函数参数中的配置对象 | 假设 SDK 返回结构稳定，没做防御性检查（实例：harmonyHttpAdapter 中 `data.header` 访问 — 如果 SDK 版本不同、返回结构有差异，代码直接崩溃；`data.result` 类型假设为 string / ArrayBuffer / Object，但实际 SDK 可能返回其他类型） |
| **R7** | **alias / 便捷方法 > 2 个时，逐个并排打开看内部逻辑，不只数数量。相同逻辑出现 ≥2 次 → 必须识别并标记重复** | 只确认 method 映射正确，漏看 headers / data 处理（实例：postForm / putForm / patchForm 覆盖而非合并 headers；三个方法各 15 行 header 合并逐字相同未识别） |
| **R8** | **同一类里 ≥3 个方法做同一类操作（如访问 Map、读写 header），必须验证它们使用了一致的访问模式。有例外必查原因。** | 逐方法线性审，没把同类方法横向并排比对（实例：AxiosHeaders 中 get/set/has/delete 等 10+ 个方法都调 `_findMapKey()` 再访问 Map，只有 `getSetCookie()` 直接 `_data.get(key)` — 这个例外的原因没追问） |
| **R9** | **每对语义配对的调用必须逐对校验**（subscribe/unsubscribe、addEventListener/removeEventListener、on/off、lock/unlock、open/close）：① 参数个数和类型一致；② 传递的回调引用是同一个函数对象；③ 有 add 必有 remove，有 sub 必有 unsub，数量和位置对称 | 看到单个调用合法就打勾，没逐对交叉比对（实例：`subscribe(onAbort)` 传了回调但 `unsubscribe()` 没传 → listener 泄漏；`addEventListener('abort', onAbort)` 注册的回调 ≠ `removeEventListener('abort', doAbort)` 移除的回调 → 取消逻辑静默断裂） |
| **R10** | **原版能做、移植版做不了的，必须在代码中标注平台限制注释** | 行为偏差无文档，后续接手者误判为 bug（实例：statusText 硬编码为空字符串但无说明） |

> R17-R18 属于本层（方法论层）的跨文件扩展规则，紧接 R10 定义。R11-R16 见下方"执行纪律层（防错误判定）"和"执行纪律层（注意力防跳）"两小节——分节是为了按认知任务分组，编号因此不连续。

| **R17** | **跨文件同名模块级变量（Map / Set / 单例对象）被 ≥2 个文件 import 时，必须验证 import 链解析到同一个实例。** grep 出所有 `new Map()` / `new Set()` / `new ClassName()` 实例化点，同名逻辑单例出现 ≥2 次独立 `new` → 报告 | 逐文件审各自的代码，未跨文件比对同名变量的实例化点（实例：两个文件各自 `new Map()` 存 locale → Ls 双实例，`registerLocale()` 写入的 Map 和 `import { Ls }` 拿到的不是同一个对象） |
| **R18** | **对跨文件传递的配置对象，追踪每个字段从写入点到至少一个读取点。** 写入但无读取 = 死数据（功能缺失）；读取但无写入 = 运行时 undefined（bug）。必须跨所有文件追踪完整链路，不止于当前文件 | 审了写入点也审了读取点，但未跨文件追踪字段级数据流（实例：`cfg.offset` 在 dayjs.ets:65 写入 → InternalCfg 接口缺 offset 字段 → CfgReader 无 getter → 构造函数不恢复，三步链路断裂但每一步单独看都"没问题"） |

### 执行纪律层（防错误判定）

| # | 规则 | 违反会导致 |
|---|------|-----------|
| **R11** | **任何"X 被 Y 调用"的断言，先 `grep -r` 确认。0 次引用 = 死代码，不给"应该是被用了"找理由** | 凭空编造调用关系（实例：composeSignals — 代码完整但 0 次 import，却写"由 axios 调用链使用"） |
| **R12** | **说"SDK 不支持 X"之前，先贴具体文档 / 源码段落。查不到就写"不确定，需确认"** | 用推理代替查证，把可实现功能判死（实例：onProgress 推断"回调式=无中间事件"但没查 NetworkKit 文档；maxContentLength 跟 maxRedirects 打包归类但实现路径完全不同） |

### 执行纪律层（注意力防跳 — 三 Pass 驱动）

| # | 规则 | 违反会导致 |
|---|------|-----------|
| **R13** | **注释声称的 ≠ 代码实际的。看到"已移除/已处理/已优化"注释，必须逐行验证该注释覆盖的每一条路径确实处理干净了。不信任注释。** | 注释说"已移除 null/boolean 检查"就跳过验证，未发现第二个 `isNaN(number)` 仍是死代码 |
| **R14** | **纯算术/纯计算文件同样逐行审计，不做"低风险"假设。** 一行代码如 `toInteger(days) / 7` 必须追问：返回值类型对吗？需要截断吗？精度够吗？溢出可能吗？ | 纯计算函数被大脑标记为"不会出错"而跳过，未发现返回值是浮点数而原库预期整数 |
| **R15** | **命名相似或功能对称的函数必须并排打开对比。** 不只对比核心逻辑，也对比错误处理分支、边界守卫、循环控制流（如 continue vs break）。 | 逐函数线性审计，closestTo 用 continue 而 closestIndexTo 用 break 的差异未被发现 |
| **R16** | **Pass 2 结构审计不关注逻辑正确性。** Pass 2 的唯一 KPI 是重复代码、死代码、类型逃逸、命名一致性、注释信任验证、并排对比。逻辑正确性由 Pass 1 负责。两个 Pass 的关注范围不重叠。Pass 2 若发现逻辑问题 → 报告但不判定，交给 Pass 1 定夺。 | Pass 2 审逻辑 → 和 Pass 1 重复劳作，且两个认知任务互相干扰，各自都审不深 |

**补充**：不可将构建产物（`build/`、`.hvigor/`、`.idea/`）、缓存文件、IDE 配置文件当作源码审计对象。审计范围严格限定在源文件。

**⚠️ 构建产物 ≠ 构建配置**：`hvigorfile.ts`、`hvigorfile.js`、`*-hvigorfile.ts`、`hvigor-config.json5` 等构建配置文件是**源码**，不是构建产物，必须纳入审计范围。项目根目录下所有非产物的 `*.ts` / `*.js` / `*.json5` / `*.json` 配置文件，无论是否由脚手架生成，均须逐文件审计。脚手架生成的代码同样是代码——硬编码的绝对路径、错误的插件版本、缺失的 task 注册，都是真实 bug。

审计前先确认 `ls` 列出的是源文件而非编译缓存或 IDE 生成物。

评判标准 ①-⑥ 是执行纪律 R1-R18 的上层指导：标准定义"目标（WHAT）"，纪律定义"手段（HOW）"。
R2 是 R1 的第二跳 — R1 确认调用链存在，R2 确认调用链的接口契约匹配。两者合在一起才是完整的调用链审计。
R4 是 R3 的输入空间扩展 — R3 比对类型声明与消费代码，R4 比对原版输入空间与本版覆盖。两者合在一起覆盖"声明了什么"和"实际覆盖了多少"。
R6 是 R5 的防御前置 — R5 管"崩了之后有没有人接"（catch），R6 管"崩之前有没有人防"（守卫）。两者合在一起构成完整的防御深度。
R8 是 R7 的横向扩展 — R7 管 alias 方法的并排比对，R8 管同一个类里同类操作的一致性比对。合在一起覆盖所有需要横向比对的方法组。
R9 是 R5 的配对扩展 — R5 管 try/catch 和 if/else 的成对审计，R9 管 subscribe/unsubscribe 和 add/remove 的成对审计。合在一起覆盖代码中所有需要对称检查的语义配对。
R11 是 R1 的执行层镜像 — R1 定义"应该问什么"，R11 定义"不许用推理代替答案"。
R12 是 R10 的执行层镜像 — R10 管"确认是限制后的标注"，R12 管"判定它是限制之前的证据门槛"。
R14 是 R4 的注意力补丁 — R4 管"输入空间覆盖全不全"，R14 管"纯计算文件有没有因为看起来简单而被跳过"。
R15 是 R8 的横向扩展 — R8 管同类方法的一致性比对，R15 管命名相似函数的并排比对（含错误分支和控制流）。
R13 是 R11/R12 的同类 — R11 管"不看证据就编造调用关系"，R13 管"不看代码就信注释"。
R16 是架构纪律 — 确保三个 Pass 不互相覆盖同一件事，各自有唯一 KPI。
R17 是 R1/R2 的跨文件扩展 — R1 管"函数被谁调用"，R2 管"调用链契约匹配"，R17 管"跨文件的同名变量是否指向同一个实例"。合在一起覆盖调用关系和实例完整性。
R18 是 R3 的跨文件扩展 — R3 管"类型声明与消费代码双向比对"（单文件内），R18 管"字段写入-读取链路追踪"（跨文件）。合在一起覆盖数据流闭环。

### AI 自审计盲区清单（Group B 独有 — 从历史 session 内部 vs 外部审计差异中提取）

> ⚠️ 此清单是 Group B 独有的反模式知识积累。AI 若不读此节，在审计中会系统性地漏掉以下 10 类问题——因为这些问题不在 AI 的"常识性审计"范围内，而是从实际 session 的漏判中归纳出来的。

| # | 盲区 | 为什么 AI 容易漏 | 已在哪些 session 中被外部审计捕获 |
|---|------|------------------|----------------------------------|
| **B1** | **纯计算函数的返回值类型** | AI 看到算术操作就打勾，不追问浮点数/整数/溢出 | date-fns: conversion.ets 返回 `number` 但原库预期 `integer` |
| **B2** | **注释声明的"已处理"未逐行验证** | AI 信任注释内容，不实际 grep 查看代码路径 | date-fns: "已移除 null/boolean 检查" 注释下仍有 `isNaN(number)` 死代码 |
| **B3** | **相似函数的控制流差异** | AI 逐函数审计，不做并排对比 | date-fns: `closestTo` 用 `continue` 但 `closestIndexTo` 用 `break` |
| **B4** | **跨文件重复代码** | AI 单文件审计，不横向比对不同文件中的相同逻辑 | date-fns: 6 个 RegExp + `cleanEscapedString` 在 2 个文件中重复 |
| **B5** | **跨文件同名模块级变量被重复实例化** | AI 不在不同文件的 `new Map()` / `new Set()` 之间做跨文件比对 | dayjs: `Ls` 在两个文件中各自 `new Map()` → 双实例 bug |
| **B6** | **跨文件字段写入-读取链路断裂** | AI 审了写入点也审了读取点，但不跨文件追踪字段级数据流 | dayjs: `cfg.offset` 写入 → `InternalCfg` 接口缺字段 → `CfgReader` 无 getter |
| **B7** | **低风险文件整文件跳过** | AI 将"看起来简单"的文件（如纯常量、纯配置）标记为低风险而跳过 | 多次出现：`constants.ets`、`hvigorfile.ts` 被跳过但含硬编码路径 |
| **B8** | **安全边界的"应该安全"假设** | AI 看到外部输入点（URL/header/body），但假设"这条路径应该安全"而跳过安全审计 | semver: `parse()` 中 regex 输入无长度限制 → ReDoS 向量未被内部审计发现 |
| **B9** | **错误分支的对称性** | AI 审了 `try`/`if` 成功分支，不同步审查对应 `catch`/`else` | axios: headers `.toJSON()` 不对称 + rejected 拦截器链 3 处缺失 |
| **B10** | **Platform API 假设的过时版本信息** | AI 从训练数据中回忆 HarmonyOS API 行为，但实际 SDK 版本行为已改变 | animate.css: `TransitionEffect` API 行为在 6.1.1(24) 中与训练数据不一致 |

**防盲区强制操作（每轮审计必须执行）**：

```
🔍 盲区扫描 — 逐项确认：
- [ ] B1: 扫描所有纯算术/纯计算文件 → 逐函数验证返回值类型、精度、溢出（R14 驱动）
- [ ] B2: grep "已移除\|已处理\|已优化\|已修复" → 对每个匹配行，打开所在文件逐路径验证
- [ ] B3: 列出所有命名相似/功能对称的函数对 → 并排打开对比控制流（含 break/continue/return 差异）
- [ ] B4: grep 相同代码块（正则/helper 函数）→ 跨文件去重检查
- [ ] B5: grep "new Map\|new Set\|new ClassName" → 同名逻辑单例出现 ≥2 次独立 new → 报告
- [ ] B6: 列出所有跨文件传递的配置对象 → 追踪每个字段从写入点到至少一个读取点
- [ ] B7: 确认所有源文件（含常量和配置文件）均已打开审阅，无一标记为"低风险跳过"
- [ ] B8: 列出所有外部输入点 → 逐点验证安全边界（SSRF/注入/大小限制/类型检查）
- [ ] B9: 列出所有 try/catch 和 if/else → 逐对验证对称性（catch 块非空、else 分支存在）
- [ ] B10: 列出所有 HarmonyOS SDK API 调用 → 逐条对照当前项目 SDK 版本（非训练数据版本）
```

## 一、为什么要双层审计

单层审计存在盲区：

| 审计方式 | 核心问题 | 盲区 |
|---------|---------|------|
| **Layer 1 — 源库对齐** | "移植版和原库行为一致吗？" | 原库本身可能不安全；移植过程中引入了新攻击面 |
| **Layer 2 — 独立对抗** | "不考虑原库，这段代码能被怎么利用？" | 可能报告与原库一致但无关紧要的差异 |

**两个 Layer 必须独立执行**，最后合并去重产出统一问题清单。

> ⚠️ **术语注意**：本文 L1（Layer 1 源库对齐审计）/ L2（Layer 2 独立对抗审计）与 `04-test-generator.md` 中的 L1-L5（测试覆盖层级：正常路径/边界条件/异常路径/启动路径/参数矩阵）含义完全不同。两组术语在各自文档内独立使用，不可交叉理解。

## 二、Layer 1 — 源库对齐审计（Fidelity Audit）

### 2.1 目标

找出移植版与原库的**行为偏差**——API 签名、返回值类型、边界条件、默认值、调用范式。

### 2.2 参照物

- 原库的官方文档 + 源码（GitHub）
- 原库的类型定义（`.d.ts`）
- 原库的测试用例（如可用）

### 2.3 检查维度

| 维度 | 检查内容 | 方法 |
|------|---------|------|
| **API 覆盖率** | 原库有多少公开 API？移植版实现了多少？缺失了哪些？ | API 清单对比 |
| **函数签名** | 参数数量、类型、可选性是否一致？ | 逐函数对比签名 |
| **返回值类型** | 返回类型是否匹配？（特别关注 Map vs Record、Promise vs callback） | 类型标注对比 |
| **默认值** | 可选参数的默认值是否和原库一致？ | 逐参数检查 |
| **边界条件** | 空输入、null/undefined、负数、极大/极小值的处理是否一致？ | 边界值测试 |
| **副作用** | 是否修改了调用方传入的引用类型参数？（原库不应有的副作用） | 逐函数检查可变性 |
| **调用范式** | 链式调用、配置对象、回调 — 使用方式是否一致？ | API 风格对比 |
| **输入空间覆盖** | 原版对每个参数支持哪些子类型 / 嵌套结构？移植版是否全覆盖？标注收窄项 | 逐参数对比输入空间（特别关注 object / Array / union 类型参数） |
| **错误处理** | 异常类型、错误消息、错误码是否对齐？ | 错误路径对比 |

### 2.4 产出

Layer 1 问题清单，每条标注：
- 文件:行号
- 原库行为
- 移植版行为
- 偏差严重程度（Critical/High/Medium/Low）
- 修复建议

### 2.5 判定准则

> **关键原则：Layer 1 只报告偏差，不评价原库设计。**
>
> 如果原库 `isAbsoluteURL` 允许 `file://` 协议，移植版也允许 → Layer 1 **不报告**。这个"允许 file://"的问题由 Layer 2 来抓。

## 三、Layer 2 — 独立安全/质量审计（Adversarial Audit）

### 3.1 目标

**完全抛开原库**，将移植版视作一个独立的新项目，从攻击者视角审计。

### 3.2 核心原则

1. **不拿原库当安全基准** — 原库运行在浏览器沙箱/Node.js，移植版运行在 HarmonyOS，攻击面不同
2. **假设调用方不可信** — 任何外部输入（URL、headers、参数、配置对象）都是潜在攻击向量
3. **追踪数据流全链路** — 输入 → 处理 → 存储 → 输出，每个环节检查

### 3.3 安全性检查维度

| 类别 | 检查项 | 参考标准 |
|------|--------|---------|
| **SSRF** | URL/IP 过滤、协议限制、路径遍历、DNS rebinding | OWASP API2:2023 |
| **注入** | CRLF 注入、header 注入、SQL/命令注入、模板注入 | CWE-74 / CWE-113 |
| **凭据泄露** | 日志中是否打印 token/password/key？序列化是否脱敏？ | CWE-532 |
| **资源耗尽** | 输入大小无上限（OOM）、递归无深度限制（栈溢出）、ReDoS | CWE-400 / CWE-1333 |
| **原型污染** | `__proto__`/`constructor`/`prototype` 键过滤、Map→Object 序列化路径 | CWE-1321 |
| **加密/随机数** | `Math.random()` 是否用于安全场景？TLS 配置是否可调？ | CWE-338 |
| **供应链** | 依赖项的安全状况、是否有过度依赖 | OWASP A06:2021 |
| **取消/中断** | signal 链是否完整？资源清理是否可靠？unsubscribe 是否被覆盖？ | — |

### 3.4 代码质量检查维度

| 类别 | 检查项 |
|------|--------|
| **重复代码** | 同一函数/逻辑是否在多处复制粘贴？ |
| **死代码** | 定义但从未使用的变量/函数/参数 |
| **类型安全** | `any`/`Object` 逃逸、类型断言是否正确、类型导出是否完整 |
| **错误处理** | try-catch 覆盖、错误传播链、无静默吞错 |
| **边界条件** | null/undefined 检查、数组越界、除零保护 |
| **命名一致性** | 文件名、函数名、变量名风格是否统一 |
| **测试充分性** | 测试是否覆盖关键路径？是否包含失败 / 边界场景？是否在真实平台运行（而非仅 smoke test）？ |

### 3.5 性能检查维度

| 类别 | 检查项 |
|------|--------|
| **算法复杂度** | O(n²) 应 O(n)？O(n) 遍历应用 O(1) 查找？ |
| **内存分配** | 不必要的对象创建、GC 压力、内存泄漏（事件监听未清理） |
| **连接管理** | HTTP 连接复用？TLS 握手是否每次重建？ |
| **动画性能** | GPU 层属性使用、不可见阶段的浪费计算 |

### 3.6 产出

Layer 2 问题清单，每条标注：
- 文件:行号
- 攻击面/质量缺陷描述
- 利用场景（如是安全漏洞）
- 严重程度（Critical/High/Medium/Low）
- 修复建议

## 四、严重程度判定标准

| 级别 | 安全 | 功能 | 质量 |
|------|------|------|------|
| 🔴 **Critical** | 可直接利用导致数据泄露/代码执行/拒绝服务 | 核心功能静默失效或返回错误结果 | N/A（Quality 问题至多 Medium——质量问题不影响安全性或功能正确性） |
| 🟠 **High** | 存在利用路径但需特定条件 | 重要功能行为偏差，用户可能注意不到 | 影响可维护性/可读性的系统性问题 |
| 🟡 **Medium** | 攻击面存在但利用困难 | 边缘功能偏差 | 单文件内的局部问题 |
| 🔵 **Low** | 信息泄露或最佳实践偏离 | 几乎无影响的差异 | 代码风格建议 |

### 5.3 AI 解析规则

AI 收到回复后按以下规则解析并生成修复队列：

- `✅` 单个问题 → 加入修复队列
- `✅` 整个级别 → 该级别全部加入修复队列
- `❌` → 从队列剔除，记录驳回原因
- `⏭️ ...` → 标注平台限制（SDK / ArkTS 不支持，无法修复），记录说明
- `📎 ...` → 标注功能收窄（有意不做，设计决策），记录原因
- `📌` → 加入「后续提醒」列表，Phase 6 前再次提示
- 未提及的问题编号 → 默认按分层建议处理（P0/P1 视为待确认，P2/P3 视为 📌）

### 5.4 人工审核要点

1. **多源同时报的**（同一问题被多个来源独立发现）：真实问题概率极高，优先修复。特别关注 L1 和 L2 同时发现的、阶段一和阶段二同时发现的
2. **仅 L1 报的**：判断是 ⏭️ 平台限制（如 ArkTS R5/R9/R14/R16 导致的必然偏离）还是 📎 功能收窄（有意简化，如 buildURL 不支持嵌套对象）
3. **仅 L2 报的**：检查是否是原库本身的问题（移植版继承了原库的安全缺陷）
4. **仅阶段二（P1/P2/P3）新发现的**：阶段一 L1/L2 未发现但三 Pass 抓到了 → 验证是否因注意力分配不同而漏掉（通常质量较高）
5. **不确定的**：标记为 "待验证"，写最小 POC 验证

### 6.1 子循环 A：修复-重建-重审（对应 05-runbook Loop 2）

子循环 A 本身是**嵌套循环**结构：

## 八、三 Pass 窄聚焦深度审计

双层审计（§一 ~ §三）保证覆盖面 — L1 对齐原库，L2 独立对抗，两个互补视角不会漏看大类问题。但单次全量扫描存在注意力缺陷：

- "看起来简单"的文件/函数被加速跳过（如纯算术 conversion.ets）
- 注释声称"已处理"就信任了，未逐行验证（如 "已移除 null/boolean 检查"）
- 相似函数各审各的，未并排对比（如 closestTo vs closestIndexTo）
- 一个 AI 同时关注"逻辑对不对"和"代码脏不脏"，两个认知任务互相干扰

**三 Pass 不是替代 L1/L2，是叠加补充。** 两者是审计流程的两个阶段，A + B 的覆盖 > 各自单独跑。

### 8.1 2×3 架构总览

```
                    Pass 1 语义            Pass 2 结构            Pass 3 补漏
                    "行为对吗？"           "代码干净吗？"          "前两轮漏了什么？"
  ──────────────────────────────────────────────────────────────────────────────────
  L1 源库对齐    │  API签名/返回值/      │  移植引入的重复/       │  P1+P2 覆盖了源库
                 │  边界/副作用/         │  冗余/文件命名         │  对齐，还有什么
                 │  默认值/错误处理      │  偏离原库              │  行为/结构偏差？
  ──────────────────────────────────────────────────────────────────────────────────
  L2 独立对抗    │  安全漏洞/注入/       │  死代码/注释欺骗/      │  P1+P2 覆盖了独立
                 │  凭据泄露/原型污染/   │  类型逃逸/命名/        │  对抗，还有什么
                 │  资源耗尽/Signal链/   │  并排对比不一致/       │  安全/质量/性能
                 │  算法复杂度/连接复用  │  不必要分配/内存泄漏   │  问题？
  ──────────────────────────────────────────────────────────────────────────────────
  兜底          │  "不确定算不算 → 报"   │  "不确定算不算 → 报"   │  "0发现≠完美，
                 │                       │                        │  连续2轮0=收敛"
```

每个格子有唯一关注点，不重叠。

### 8.2 执行调度表（映射到 §七 检查清单）

三 Pass 不重写 §七 的检查清单，而是从完整清单中选取各自负责的项：

| Pass | 取自清单 | 关注 | 不关注 |
|------|---------|------|--------|
| **P1 语义** | 7.1 L1 全部（API/签名/返回值/边界/副作用/错误） | 行为一致吗？ | 命名、重复、代码风格 |
| | 7.2 L2 安全全部（SSRF/注入/凭据/原型污染/Signal链） | 逻辑有漏洞吗？ | — |
| | 7.4 L2 性能-算法项（复杂度/连接复用） | 算法对吗？ | 分配模式、GC |
| | 7.6 SDK 假设项 | SDK 行为假设成立吗？ | — |
| **P2 结构** | 7.1 L1-结构项（项目结构/构建配置） | 原库没有的冗余/偏离 | 签名正确性 |
| | 7.3 L2 质量全部（重复/死代码/类型/命名） | 代码干净吗？ | 逻辑正确性 |
| | 7.4 L2 性能-模式项（分配/内存泄漏/事件清理） | 模式好吗？ | 算法复杂度 |
| | 7.5 文档/测试/元数据 | 文档齐全、测试到位 | — |

| **P3 补漏** | 7.1 ~ 7.6 全部 | P1+P2 漏了什么？ | — |

### 8.3 各 Pass 执行细则

#### Pass 1 — 语义审计

```
关注：代码的行为是否正确
不关注：命名好不好、有没有重复代码、类型逃逸（这些是 P2 的责任）

输入：
  - ArkTS 源码（全部）
  - 原 JS 源码（L1 对比用）
  - §七 检查清单中 8.2 调度表标注"P1"的项

方法：
  - 逐函数对比 JS vs ArkTS 的输入空间、输出类型、边界行为、副作用
  - 对每个外部输入点追踪数据流全链路
  - R14（不跳过纯计算）：纯算术函数同样逐行验证返回值类型、精度、溢出

产出：语义问题清单
  每条标注 [L1-P1] 或 [L2-P1]

兜底：以上调度表是启发式，不是穷举。任何"行为不一致"的发现，
      无论是否匹配调度表中的具体项，均须报告。不确定算不算 → 报。
```

#### Pass 2 — 结构审计

```
关注：代码是否干净、无冗余、无死代码
不关注：逻辑是否正确（这是 P1 的责任，不是 P2 的）
不信任：注释声明（R13 驱动）

输入：
  - ArkTS 源码（全部）
  - 原 JS 源码（L1-P2 对比用——原库有这个重复吗？移植多出来的？）
  - §七 检查清单中 8.2 调度表标注"P2"的项
  - L1 + L2 阶段的问题清单 A（参考：避免重复报告已发现的项）

专用纪律（R13-R16）：
  R13 不信任注释  — 看到"已移除/已处理/已优化"注释，逐行验证处理干净了
  R14 不跳过纯计算 — 纯算术文件同样逐行审计，不做"低风险"假设
  R15 并排对比    — 命名相似/功能对称的函数，打开并排对比（含错误分支和控制流）
  R16 不关注逻辑  — P2 的唯一 KPI 是结构问题。若发现逻辑问题 → 报告但不判定，交给 P1 定夺

强制操作：
  - 机械列出全部源文件路径，逐个打勾已审 ✓（格式见 8.4）。缺一个 → P2 不算完成
  - 对 ≥2 处出现的相同代码块/正则/helper，标注重复及出现位置

产出：结构问题清单
  每条标注 [L1-P2] 或 [L2-P2]

兜底：以上调度表是历次审计遗漏模式的编码，不是穷举。任何"代码可以
      写得更干净"的发现均须报告。不确定算不算 → 报。
```

#### Pass 3 — 补漏

```
关注：P1 + P2 漏了什么？
不关注：无。P3 的唯一 KPI 是找到前两轮的遗漏

输入：
  - P1 清单 + P2 清单（前两轮的产出）
  - L1 + L2 阶段的问题清单 A（逐项对比：A 里的每一项 P1+P2 都覆盖确认了吗？）
  - 已知盲区模式列表（从历次迁移/外部审计中积累，持续追加）：
    · "纯计算文件被跳过（返回值类型/精度/溢出未验证）"
    · "注释声称已处理但未逐行验证"
    · "相似函数未并排对比（含错误分支和控制流差异）"
    · "跨文件重复代码（相同 helper/正则 在多处定义）"
    · "简单文件全文件跳过（未打开就判定为低风险）"
    · "跨文件同名变量/常量被重复实例化（相同逻辑单例出现 ≥2 个独立实例，如两个文件各自 new Map() 存同一个逻辑状态）"
    · "跨文件字段写入-读取链路断裂（写入的字段在消费端接口/读取路径中缺失，导致静默丢弃）"
    · "安全边界假设未验证（B8：API 权限检查 / 输入净化 / SSR 防护假设被误判为已满足）"
    · "错误分支对称性漏检（B9：正常路径改了但异常路径未同步——try/catch 只改了 try 块，catch 块静默吞错误）"
    · "平台 API 版本边界假设（B10：使用了某 API 版本才有的特性但未检查 @version 注解，低版本设备上运行时崩溃）"
  - §七 检查清单 7.1 ~ 7.6 全部

方法：
  - 逐项对比清单 A → 有没有 A 发现了但 P1/P2 没覆盖到的？
  - 对已知盲区模式做定向扫描（每个模式逐个文件夹验证）
  - 自由扫描：不管归属哪个维度，"还有什么不对的？"

产出：补充发现清单（通常很少，但质量高）
  每条标注 [L1-P3] 或 [L2-P3]

收敛判定：连续两轮 P3 均为 0 发现 → 可判定为收敛
兜底：0 发现不代表完美，只代表"这一轮没找到"。
```

### 8.4 Pass 2 全文件打勾表（防跳过）

### 增量审计（子循环 A 修复后重审，强制步骤）

> ⚠️ 修复完成后不得跳过此步骤直接进入下一 Phase。必须执行增量重审并产出结果。

```
AI 执行：
1. 仅加载本次修改的文件（完整源码）
2. 对修改文件的完整内容执行：
   - L1 + L2 快速重审
   - P2 结构重审（含全文件打勾表、R13 注释信任验证、R15 并排对比）
3. 输出重审结果（格式见下）
4. 新发现 → 按严重度分流：
   Critical/High → 追加到问题清单 → 人工审核
     → 有 ✅ 采纳 → 回到步骤 1（再修 → 再重审）
     → 无 ✅ 采纳 → 子循环 A 可退出
   Medium/Low → 输出清单 → 人工定夺
     → 改 → 选取 ✅ 项 → 回到步骤 1（再修 → 再重审）；未选记残余
     → 不改 → 全部记残余 → 子循环 A 可退出
5. 已修复项 → 标记完成
6. 无任何新发现 → 子循环 A 可退出
```

重审输出格式：
```
🔍 增量重审 — 修改文件: [列表]
   新 Critical: [N 项 / 无]
   新 High:     [N 项 / 无]
   新 Medium:   [N 项]
   新 Low:      [N 项]
   判定: [有新 Critical/High → 回到人工审核 / 有 Medium/Low → 人工定夺 / 无 → 退出循环]
```
